# Execution Plan — Issue #258 郵件編輯器編輯區被左右遮蔽

## 需求摘要

Power Course 後台的自動化郵件模板編輯器使用 `j7-easy-email-extensions` 的 `StandardLayout`（左：元件區 / 中：信件畫布 / 右：屬性區）三欄佈局。該元件原為「整頁全螢幕」設計，被嵌入 WordPress 後台後，外層還疊著 WordPress 主選單 + Refine.dev 側邊欄，多層側邊欄擠壓下，中間信件畫布變得又窄又扁，管理員必須反覆左右捲動才能拼湊出信件全貌，無法正確排版與確認最終呈現效果。

本次為**純前端版面（layout / CSS / 容器層）呈現優化**，**不涉及 API、不涉及資料模型變更**，也不改變任何已儲存信件的實際內容或寄出排版。

## 澄清結論（Issue #258 留言 `D C A A C B`）

| # | 決策 | 選項 | 內容 |
|---|------|------|------|
| Q1 | 修正主要方向 | D | 沉浸全螢幕（隱藏 WP 主選單 + Refine 側邊欄，三欄編輯器獨佔瀏覽器寬度）+ 面板可各自收合 |
| Q2 | 進入沉浸模式方式 | C | 進入編輯頁自動進入沉浸模式；提供切換按鈕可自由來回；記住上次選擇（偏好記憶，client-side localStorage） |
| Q3 | 小螢幕面板行為 | A | 窄斷點時左右面板改「浮動抽屜」——平常收起，點按鈕滑出覆蓋在畫布上方，不佔用畫布寬度 |
| Q4 | 中間畫布最小寬度 | A | 以信件標準 600px 為基準，中間畫布可視寬度最少 **640px**（含左右內距），不足時才觸發面板收合／橫向捲動 |
| Q5 | 實作範圍 | C | 交由實作判斷，優先低風險 CSS／容器層覆寫（不動套件本身），必要時再自訂佈局；不破壞既有儲存邏輯 |
| Q6 | 驗收螢幕尺寸 | B | 四斷點：13 吋（~1280）＋ 14/15 吋（~1440）＋ 桌機（1920）＋ 超寬（2560） |

## 驗收標準（對齊 Issue #258）

1. 進入郵件模板編輯頁時，中間信件畫布完整顯示、不被左右面板遮蔽或切掉。
2. 編輯過程中，左側元件區、中間畫布、右側屬性區三者不互相重疊遮蔽，皆可正常操作。
3. 管理員不需為了看見信件全貌而反覆左右捲動（中間畫布可視寬度 ≥ 640px）。
4. 四斷點（1280 / 1440 / 1920 / 2560）下，中間畫布都維持足以閱讀與編輯的寬度。
5. 退出沉浸模式後，不影響 WordPress 後台既有的選單、工具列與儲存按鈕的正常顯示與操作。
6. 已儲存信件的實際內容與寄出排版維持不變（僅編輯畫面呈現優化）。
